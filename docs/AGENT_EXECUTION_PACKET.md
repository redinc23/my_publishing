## 0. Read this before dispatching anything

Three facts govern every decision below. They were verified against the repository
at the baseline SHA, not taken from any document's self-description.

**Fact 1 — The codebase is further along than the ledger implies.** 338 TypeScript
files, ~39,000 lines, 62 page routes, 29 API routes, 47 test specs, Playwright + k6
harnesses, Sentry wired on client/server/edge, CI green on main, and production
answering 200 at `mangu-publishers.com`. Most Product Gap items are **state-completion
and truth-correction on existing surfaces**, not net-new construction. Any agent that
starts building from scratch has misread the repo.

**Fact 2 — The blocking work is not code.** `docs/NEXT_GO.md` records G13 TRUE,
G12 PARTIAL, G1–G11 FALSE, Phases 11–16 NOT STARTED, and every row of
`docs/OPERATOR_QA_LOG.md` blank. The gates that remain FALSE are blocked on
environment promotion, live-backend QA, Stripe end-to-end verification, DNS cutover,
and legal approval — human-gated activities. Adding agent throughput does not move them.

**Fact 3 — Governance overhead is now the main efficiency risk.** The repo holds 40+
docs and 24 skill packs for one solo operator. Every additional crosswalk column has a
carrying cost. This packet therefore **cuts** work from the handover plan wherever the
cut does not weaken evidence: see §5.

## 1. The efficiency thesis

The handover document proposes ten bootstrap tasks (GAP-BOOT-01 … 10) followed by
nine work packages. Executed literally, that is ~19 agent runs before a single user-visible
defect is fixed. Most of those runs read the same files and re-derive the same state.

The efficient decomposition collapses them into **six runs**, because:

- GAP-BOOT-01 through 04 are all edits to one artifact (the ledger). One agent, one file,
  one PR. Splitting them quadruples the context-loading cost and produces four PRs that
  must be merged in order anyway.
- GAP-BOOT-05 through 08 are all products of a single crawl of the codebase and the
  deployed site. Route manifest, CTA ledger, and false-success register are three
  **views of one dataset**. Generating them in one pass is roughly a third of the cost
  of three passes, and guarantees they agree with each other.
- GAP-BOOT-09 and 10 are a five-minute reconciliation once 01–08 exist. They fold into
  the run that consumes them.
- The route manifest should be **generated by a script, not read by an LLM**. A crawl of
  62 pages and 29 API routes is deterministic file-system work. Spending model context on
  it is the single largest avoidable cost in the entire plan. Run 2 therefore ships a
  script first, then reasons only about its output.

Net: **6 runs instead of 19**, with the same evidence output.

### The six runs

| Run | Name | Agent | Mutates code? | Gate impact | Est. |
| --- | --- | --- | --- | --- | --- |
| R1 | Ledger normalization + crosswalk + evidence rules | A0 | No — docs only | Feeds G12 | 1 session |
| R2 | Route/CTA/truth audit (script-generated) | A1 | Script + docs only | Feeds G6 | 1 session |
| R3 | L0 truth fixes — placeholder honesty sweep | A1→A6 | Yes, narrow | **G6** | 1–2 sessions |
| R4 | Auth journey completion (Supabase, no cutover) | A2 | Yes | **G3, G5** | 2–3 sessions |
| R5 | Commerce spine E2E — purchase→webhook→library→read | A3+A5 | Yes | **G4, G8** | 2–3 sessions |
| R6 | Trust evidence — a11y, mobile, legal, SEO | A6 | Yes, narrow | **G6, G10** | 2 sessions |

Runs R1–R2 are freeze-legal today and can execute back-to-back in one working day —
this is the honest scope of the "one day" goal. R3 onward are gated on human unblocks
tracked in `HUMAN_TASKS.md`.

**Run R5 is the highest-leverage single unit of work in the repository.** It exercises
auth, RBAC, Stripe, webhooks, the order model, entitlements, the library, and the reader
in one traced path. It moves two hard gates and validates more of the stack than any
other slice. If only one thing gets done, do R5.

  Never mark a PR ready; the orchestrator controls readiness.

## 5. What this packet deliberately cuts

Efficiency comes as much from removing work as from sequencing it. These cuts were made
against the handover plan, with reasons:

| Cut | Why |
| --- | --- |
| 60 individual GitHub issues | The repo already has a feature-story backlog and an enhancement ledger. Issues are created per **slice**, only when audited, deduplicated, and dependency-ready. |
| Separate PRs for GAP-BOOT-01…04 | All four edit one artifact. Four PRs means four context loads and a forced merge order. |
| Hand-crawled route audit | Deterministic file-system work. Script it once; every later run re-runs the script instead of re-reading 62 files. |
| A standalone WP-08 planning run | Post-GO items need one YAML status (`DEFERRED`) and nothing else until GO. Planning them now is pure carrying cost. |
| A second migration plan | `docs/PROJECT_PHOENIX.md` already governs P-059. Cross-reference it. A competing plan is worse than no plan. |
| Full reviews / audiobooks / payouts | Out of launch scope per NEXT_GO.md. Honest-unavailable states satisfy the ledger; building the products is a freeze violation. |
| Mockups for every screen | Mock only surfaces R2 classifies MISLEADING or PARTIAL. The REAL ones do not need redesign. |

## 6. Definition of done

An item is PASSED only with: Product Gap ID and PR link; governing authority and shipping
lane; baseline and candidate SHA; deployment/preview ID and environment; UTC timestamp and
actor; exact test command and result; expected vs. observed behavior; the role/account
state used; device/browser where relevant; sanitized visual or API evidence; request/event
IDs where correlation matters; rollout and rollback evidence for production-impacting
changes; documentation amended in the same PR; remaining human gate or residual risk; and
gate impact — moved to TRUE, remains FALSE, or N/A.

Implementation alone is not done. A green unit test alone is not done. A screenshot alone
is not done. A deployed URL alone is not done.

## 7. Honest expectation setting

**Achievable in one day:** R1 and R2. A normalized machine-readable ledger, a scripted
route and CTA manifest, a false-success register, and a ranked queue of the next legal
slices. This is a real and valuable day's output.

**Not achievable in one day, by any agent configuration:** G1–G11 turning TRUE. Those
gates are blocked on environment promotion, live-backend QA, Stripe verification against
real infrastructure, DNS cutover, and legal sign-off. `docs/NEXT_GO.md` lists Phases 11–16
as NOT STARTED, and every row of `docs/OPERATOR_QA_LOG.md` is blank. No amount of agent
parallelism substitutes for an operator with console access.

The fastest genuine path to GO is: **R1 → R2 → R3 → R5 → R4 → R6 → gate sweep**, with
`HUMAN_TASKS.md` worked in parallel by the operator. Note that R5 can be pulled ahead of
R4 if auth evidence already exists from Phase 7A — check before sequencing.

## 8. Toward a fully agent-executable repository

The long-term goal is a repo consisting of structured prompts and data models an agent
executes directly. The concrete steps, in order:

1. **Ledger as data, not prose.** R1 delivers this. Once `product-gap-ledger.yml` is
   canonical and CI-validated, agents query it instead of parsing markdown.
2. **Manifests generated, never authored.** R2 delivers this for routes and CTAs. Extend
   the pattern: every inventory in the repo should be a script's output.
3. **The issue template as the atomic prompt unit.** Section 9 of the handover already has
   the right shape. Emit it *from* the YAML rather than filling it by hand.
4. **Evidence as an append-only structured sink.** Machine-readable, keyed by candidate
   SHA, never rewritten. Gate state then becomes a query, not a judgment call.
5. **Human gates as first-class data.** This is the part no architecture removes.
   Credentials, billing, DNS, legal approval, and irreversible cutovers must be modeled
   in `HUMAN_TASKS.md` with owner, exact action, and unblock criteria — so agents route
   around them cleanly instead of stalling or, far worse, fabricating completion.

Step 5 is the one that matters most. An agent that cannot tell the difference between
"blocked on a human" and "done" will eventually report a launch that never happened.
